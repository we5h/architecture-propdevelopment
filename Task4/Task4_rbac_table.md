# Задание 4. Ролевой доступ к кластеру Kubernetes — PropDevelopment

## Обоснование ролевой модели

Роли выведены из организационной структуры компании (описание + диаграмма):

- **Разработчики** — создают и деплоят приложения, но не должны видеть секреты prod-среды
- **DevOps-инженеры** — настраивают кластер, управляют деплойментами, но не удаляют namespace и не читают секреты других команд
- **Инженеры по эксплуатации** — мониторят состояние кластера (только чтение)
- **Специалист по ИБ** — читает секреты, аудирует политики (привилегированный доступ, но без права изменений)
- **Кластерный администратор** — полный доступ, только для платформенной команды

---

## Таблица ролей RBAC

| Роль | Права роли | Группы пользователей |
| --- | --- | --- |
| `viewer` | `get`, `list`, `watch` на ресурсы: `pods`, `services`, `deployments`, `replicasets`, `endpoints`, `events`, `configmaps` (кроме секретов). Доступ только для чтения. Не может изменять, создавать или удалять ничего. | **Инженеры по эксплуатации** — следят за состоянием сервисов в runtime. **Бизнес-аналитики** — могут смотреть состояние кластера для диагностики. **Аналитики BI** — ограниченный просмотр для понимания состояния data-сервисов. |
| `developer` | `get`, `list`, `watch`, `create`, `update`, `patch` на: `pods`, `deployments`, `replicasets`, `services`, `configmaps`, `jobs`, `cronjobs`. `delete` только на `pods` (перезапуск). `get`, `list` на `events` и `logs` (kubectl logs). Без доступа к `secrets`, `clusterroles`, `nodes`. | **Разработчики** — деплой и обслуживание своих сервисов в рамках своего namespace (client-*, tenant-*, accountant-*, data-*). Каждая продуктовая команда работает только в своём namespace. |
| `devops-engineer` | Все права `developer` плюс: `create`, `update`, `patch`, `delete` на `deployments`, `statefulsets`, `daemonsets`, `ingresses`, `persistentvolumeclaims`, `configmaps`, `services`. `get`, `list` на `secrets` (только в своём namespace). `create`/`delete` на `pods`. Управление `namespaces` — только `get`, `list` (не создаёт и не удаляет namespace без cluster-admin). | **DevOps-инженеры** — есть в каждой продуктовой команде. Настраивают инфраструктуру своего домена, управляют деплойментами, конфигурируют Ingress и хранилища. |
| `security-auditor` | `get`, `list`, `watch` на **все** ресурсы кластера включая: `secrets`, `clusterroles`, `clusterrolebindings`, `rolebindings`, `serviceaccounts`, `networkpolicies`, `podsecuritypolicies`. Только чтение — создание, изменение, удаление запрещены. ClusterRole (не ограничена namespace). | **Специалист по ИБ** — единственный в компании, проводит аудит, проверяет секреты и политики доступа, готовится к аудиту безопасности. |
| `cluster-admin` | Полный доступ ко всем ресурсам кластера (`*` verbs, `*` resources). Управление nodes, namespaces, ClusterRoles, PersistentVolumes. Использование строго регламентировано — только по заявке и с аудитом действий. | **Платформенная команда / старший DevOps** — настройка самого кластера, создание namespace для новых команд, управление RBAC. В PropDevelopment — ответственный DevOps-инженер платформы (не более 2 человек). |
