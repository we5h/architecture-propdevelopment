# Проверочный лист безопасности — PropDevelopment

## Обоснование выбора разделов

В ходе анализа архитектуры PropDevelopment выделены следующие **приоритетные проблемы**:

1. **Нарушен контроль данных** — несколько несинхронизированных точек регистрации клиентов, бесконтрольные CDC-потоки между системами, отсутствие ограничений доступа к данным со стороны систем обработки. Прямое следствие — клиенты видят в личном кабинете данные других клиентов (известный инцидент).
2. **Проблемы с API партнёров** — API управляющих компаний используются без единых политик ИБ; контракты содержат ПД и позволяют партнёрам читать/менять чужие данные.
3. **Нарушен процесс передачи информации по ИБ** — специалист по ИБ один, и ему не передаются все решения/изменения из команд.
4. **Планируемое расширение** — новые BI/ML/AI-сервисы, расширение партнёрских интеграций — требуют зрелых процессов управления данными уже сейчас.
5. **Требования законодательства РФ** — обработка ПД по 152-ФЗ, интеграции с госорганами (СМЭВ).

### Выбранные разделы и приоритеты:

| Раздел | Приоритет | Обоснование |
|---|---|---|
| **I. Управление доступом** | 🔴 Высокий | auth-service-1 и Active Directory есть на диаграмме, но наличие единой политики доступа под вопросом. Инцидент с чужими данными в ЛК прямо указывает на проблемы с разграничением доступа. |
| **II. Безопасность данных** | 🔴 Высокий | Ключевая проблема компании: нет классификации, нет контроля потоков, нет Data Governance — это прямо блокирует развитие BI/ML. |
| **III. Защита инфраструктуры** | 🟡 Средний | Есть firewall на периметре, гибридная инфраструктура (on-prem + облако). Проверить сегментацию и облачную безопасность важно в контексте расширения. |
| **IV. Управление инцидентами** | 🔴 Высокий | Специалист по ИБ один, процесс передачи информации нарушен, утечка у конкурента взволновала руководство — реагирование на инциденты явно слабое место. |
| **V.1. Общие вопросы** | 🟡 Средний | Есть мобильное приложение для собственников (Android), гибридная инфраструктура — нужна проверка MDM и облачных резервных копий. |
| **V.2. Физическая безопасность** | 🟢 Низкий | Компания имеет собственный ЦОД — базовые вопросы физической защиты актуальны, но не являются первоочередными. |

> - ❌ **Нет** — явно не реализовано или описана проблема
> - ❓ **Нет данных** — информации недостаточно для однозначного ответа; требует проверки в ходе аудита

Используется ли эта возможность — неизвестно. Особенно актуально для разграничения доступа партнёров к данным разных управляющих компаний. |

| 14 | Используются ли средства централизованного контроля доступа? | ❓ Нет данных | Keycloak как централизованный IAM присутствует, но покрывает ли он все системы (в т.ч. базы данных, хранилище данных, BI) — из описания неясно. Критично проверить на аудите. |
