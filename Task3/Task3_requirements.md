# Задание 3. Требования к внешним интеграциям — Умный дом (PropDevelopment)

## 1. Контекст и область применения

Документ описывает требования к безопасности, протоколам и организации взаимодействия при интеграции PropDevelopment с партнёрской платформой **Smart Home Platform** для реализации двух сервисов:

- **Интеллектуальный домофон** — видеодомофон с биометрическим распознаванием лиц (Face ID).
- **Интеллектуальный шлагбаум** — шлагбаум с распознаванием государственных номеров автомобилей.

Оба сервиса доступны через существующее мобильное приложение собственников.

---

## 2. Архитектурное решение по организации взаимодействия

### 2.1. Принцип единой точки интеграции

Прямая связь между внутренними системами PropDevelopment (tenant-core-app) и Smart Home Platform партнёра **не допускается**. Все запросы проходят через новый компонент — **smart-home-gateway** (API Gateway / Smart Home Adapter).

**Обоснование:** одна из ключевых проблем компании — отсутствие единых политик ИБ при интеграции с партнёрами (см. проблему с API управляющих компаний). Новая интеграция должна избежать этого паттерна с самого начала.

```
Мобильное приложение
        ↓ (HTTPS / OAuth 2.0)
  Витрина сервисов собственников
        ↓ (REST / внутренняя сеть)
   tenant-core-app
        ↓ (REST / внутренняя сеть)
  smart-home-gateway  ←→  auth-service-1 (валидация токенов)
        ↓ (REST / TLS 1.3 / OAuth 2.0)
  Smart Home Platform (партнёр)
        ↓
  Домофон / Шлагбаум (устройства)
```

### 2.2. Новые компоненты PropDevelopment

| Компонент | Тип | Описание |
|---|---|---|
| **smart-home-gateway** | Container: Kotlin, Spring Cloud Gateway | API Gateway. Применяет политики ИБ, контролирует передаваемые данные, выполняет rate limiting, логирование, валидацию токенов через auth-service-1 |
| **smart-home-db** | Container: PostgreSQL | Хранит: белые списки (разрешённые лица, номера авто), конфигурации устройств, журналы событий доступа |

## 3. Требования к безопасности

### 3.1. Управление персональными данными и биометрией

| № | Требование |
|---|---|
| **3.1.1** | Биометрические данные (шаблоны лиц) **не хранятся** на стороне PropDevelopment. Их хранение и обработка — исключительно на стороне партнёра (Smart Home Platform) в рамках его инфраструктуры. |
| **3.1.2** | Передача биометрических шаблонов лиц от PropDevelopment к партнёру **запрещена**. PropDevelopment передаёт только идентификаторы собственников (tenant_id) и разрешения на доступ. |
| **3.1.3** | Номерные знаки автомобилей классифицируются как **персональные данные** (152-ФЗ). Хранение и передача номеров должны осуществляться с соблюдением требований 152-ФЗ. |
| **3.1.4** | При регистрации биометрии собственник должен дать **явное письменное согласие** (в мобильном приложении — подтверждение с сохранением факта согласия в smart-home-db). |
| **3.1.5** | Собственник должен иметь возможность **отозвать согласие** и удалить свои биометрические данные из системы партнёра через мобильное приложение. |
| **3.1.6** | Данные о событиях доступа (кто, когда, куда получил доступ) хранятся в smart-home-db **не более срока, установленного политикой хранения данных** (рекомендуется 90 дней, но не менее требований законодательства). |

### 3.2. Изоляция данных между ЖК

| № | Требование |
|---|---|
| **3.2.1** | Данные одного жилого комплекса (ЖК) **изолированы** от данных другого. Партнёр не должен иметь возможности читать или изменять данные ЖК, к которым у него нет доступа. |
| **3.2.2** | Все запросы к партнёру содержат идентификатор ЖК (complex_id). Партнёр обязан фильтровать данные по этому идентификатору. |
| **3.2.3** | smart-home-gateway **валидирует** принадлежность запрашиваемых ресурсов к ЖК аутентифицированного пользователя перед отправкой запроса партнёру. |

### 3.3. Передача данных

| № | Требование |
|---|---|
| **3.3.1** | Все соединения между smart-home-gateway и Smart Home Platform используют **TLS 1.3** (минимум TLS 1.2). Использование устаревших версий (TLS 1.0, 1.1, SSL) запрещено. |
| **3.3.2** | Сертификаты партнёра проходят **mutual TLS (mTLS)** аутентификацию — обе стороны предъявляют клиентский сертификат. |
| **3.3.3** | Используется **pinning сертификатов** на уровне smart-home-gateway для предотвращения атак типа «человек посередине». |
| **3.3.4** | В API-контрактах с партнёром **явно перечислены** все передаваемые поля. Передача полей, не описанных в контракте, блокируется на уровне gateway. |
| **3.3.5** | Партнёр **не получает** ПД клиентов PropDevelopment (ФИО, email, телефон, адрес). Взаимодействие ведётся через обезличенные идентификаторы (tenant_id, complex_id). |

### 3.4. Контроль доступа к API партнёра

| № | Требование |
|---|---|
| **3.4.1** | Применяется **rate limiting** на уровне smart-home-gateway: не более N запросов в секунду на ЖК (конкретные значения определяются при нагрузочном тестировании). |
| **3.4.2** | Все запросы к партнёру **логируются** в smart-home-db: timestamp, tenant_id, complex_id, тип запроса, код ответа. |
| **3.4.3** | В случае **аномальной активности** (резкий рост числа запросов, ошибки авторизации) smart-home-gateway генерирует алерт для службы ИБ. |
| **3.4.4** | API-ключи и секреты для аутентификации у партнёра хранятся в **защищённом хранилище секретов** (Vault или аналог), а не в конфигурационных файлах. |

### 3.5. Аудит и соответствие требованиям

| № | Требование |
|---|---|
| **3.5.1** | Интеграция проходит **проверку ИБ-специалиста** до выхода в продакшн. |
| **3.5.2** | API-контракт партнёра проверяется на предмет **избыточных прав** (право читать/изменять чужие данные) — аналогичная проблема уже выявлена с управляющими компаниями. |
| **3.5.3** | Договор с партнёром включает положения о **обработке персональных данных** (соглашение об обработке ПД по 152-ФЗ), в т.ч. биометрии и номеров авто. |
| **3.5.4** | Проводится **ежеквартальный аудит** API-контрактов и журналов событий интеграции. |

## 4. Протоколы аутентификации и авторизации

### 4.1. Аутентификация собственников (мобильное приложение → PropDevelopment)

| Параметр | Значение |
|---|---|
| Протокол | **OAuth 2.0 + OpenID Connect (OIDC)** |
| Сервис | auth-service-1 (Keycloak) |
| Тип токена | JWT (JSON Web Token) |
| Срок действия access token | 15 минут |
| Refresh token | 30 дней, ротируемый |
| MFA | Обязательна для операций управления доступом (добавление/удаление лиц, номеров авто) |

**Поток аутентификации:**
1. Мобильное приложение запрашивает токен у auth-service-1 (Authorization Code Flow + PKCE).
2. auth-service-1 возвращает JWT access token с claims: `sub` (tenant_id), `complex_id`, `roles`.
3. Мобильное приложение передаёт токен в заголовке `Authorization: Bearer <token>` при каждом запросе.
4. smart-home-gateway валидирует подпись токена через JWKS endpoint auth-service-1 (не хранит секрет локально).

### 4.2. Авторизация запросов к Smart Home Platform (PropDevelopment → партнёр)

| Параметр | Значение |
|---|---|
| Протокол | **OAuth 2.0 Client Credentials Flow** |
| Субъект | smart-home-gateway (сервисная учётная запись) |
| Scope | Ограниченный: `smarthome:read`, `smarthome:command`, `smarthome:access-control` — только необходимые разрешения |
| Срок действия service token | 1 час |
| Дополнительная защита | mTLS (mutual TLS) — клиентский сертификат со стороны smart-home-gateway |

**Поток авторизации:**
1. smart-home-gateway запрашивает service token у OAuth-сервера партнёра с использованием client_credentials и mTLS.
2. Полученный токен используется для всех запросов к партнёру в течение срока действия.
3. Токен обновляется за 5 минут до истечения срока (pre-emptive refresh).

### 4.3. Авторизация действий (управление списками доступа)

| Роль | Разрешённые действия |
|---|---|
| `owner` | Просмотр и управление своим доступом (своё лицо, свои авто) |
| `building_manager` | Управление доступом всего ЖК (все жители) |
| `admin` | Полный доступ, конфигурация устройств |

Проверка ролей выполняется в **smart-home-gateway** на основе claims из JWT-токена собственника. Партнёру роли не передаются.

## 5. Организация взаимодействия между системами

### 5.1. Операции и потоки данных

| Операция | Инициатор | Протокол | Данные |
|---|---|---|---|
| Открытие домофона вручную | Собственник (мобильное приложение) | REST → smart-home-gateway → REST партнёру | tenant_id, device_id, команда |
| Открытие домофона по биометрии | Домофон (устройство партнёра) | Устройство → Smart Home Platform (партнёр, без участия PropDev) | Биометрия обрабатывается только у партнёра |
| Добавление лица в список доступа | Собственник (мобильное приложение) | REST → smart-home-gateway → REST партнёру | tenant_id, complex_id, разрешение на регистрацию биометрии |
| Открытие шлагбаума по номеру | Шлагбаум (устройство партнёра) | Устройство → Smart Home Platform | Номер авто обрабатывается у партнёра |
| Добавление номера авто | Собственник (мобильное приложение) | REST → smart-home-gateway → REST партнёру | tenant_id, complex_id, хэш номера авто |
| Получение журнала событий | Собственник (мобильное приложение) | REST → smart-home-gateway → smart-home-db | События из локальной БД |
| Уведомление о событии | Smart Home Platform → PropDevelopment | Webhook → smart-home-gateway → tenant-core-app | event_type, device_id, complex_id, timestamp |

### 5.2. Обработка ошибок и отказоустойчивость

| Ситуация | Поведение |
|---|---|
| Smart Home Platform недоступна | smart-home-gateway возвращает 503 с понятным сообщением. Мобильное приложение отображает статус «Сервис временно недоступен». Физические устройства продолжают работу в автономном режиме на базе кешированных списков. |
| Истёк service token | smart-home-gateway автоматически обновляет токен и повторяет запрос (один retry). |
| Партнёр вернул ошибку авторизации | Алерт в службу ИБ, запись в журнал, запрос не повторяется автоматически. |
| Невалидный JWT собственника | smart-home-gateway возвращает 401, мобильное приложение инициирует повторную аутентификацию. |

### 5.3. Минимально необходимые данные в API-контракте с партнёром

PropDevelopment придерживается принципа **минимизации данных** (data minimization) в соответствии с 152-ФЗ и ISO 27001. В запросах к партнёру передаются только:

- `complex_id` — идентификатор ЖК
- `device_id` — идентификатор устройства
- `access_token_partner` — токен для авторизации запроса
- Для операций с доступом: разрешение или запрет (`grant` / `revoke`) и идентификатор пользователя PropDevelopment (не ФИО, не email)
- Для номеров авто: **только хэш номера** (SHA-256), без передачи номера в открытом виде

**Не передаётся партнёру:**
- ФИО, email, телефон собственника
- Адрес квартиры
- Данные других жильцов ЖК
- Финансовые данные
- Данные об оплате ЖКУ

## 6. Чек-лист готовности интеграции к продакшн

| № | Пункт | Ответственный |
|---|---|---|
| 1 | API-контракт партнёра проверен ИБ-специалистом на избыточные права | ИБ-специалист |
| 2 | Договор с партнёром содержит соглашение об обработке ПД (152-ФЗ) | Юридический отдел |
| 3 | mTLS настроен и протестирован | DevOps |
| 4 | smart-home-gateway прошёл нагрузочное тестирование с rate limiting | QA |
| 5 | Журналирование всех запросов к партнёру проверено | ИБ-специалист |
| 6 | Процедура отзыва биометрии собственником протестирована end-to-end | QA |
| 7 | Автономный режим устройств при недоступности партнёра проверен | QA |
| 8 | Алерты при аномальной активности настроены | DevOps / ИБ |
